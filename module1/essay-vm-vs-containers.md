Виртуализация нужна для:
1. утилизации ресурсов;
2. безопасность;
3. Удобство администрирования.

## VM
VM эмулирует hardware и запускает на нем гостевую ОС с собственным ядром.
Есть 2 вида VM:
* полная виртуализация;
* паравиртуализация.

**Полная виртуализация** - виртуализация, когда гостевая ОС не знает о том, что она виртуальная, и определяет железо как физическое.
В таком типе виртуализации используется динамическая трансляция - когда процесс внутри этой ОС выполняет системный вызов, гипервизор перехватывает этот сис. вызов и перенаправляет на реальное железо.

**Паравиртуализация** - гостевая ОС с модифицированным ядром, и она знает о том, что она виртуализированная. Связь гостевой ОС и гипервизора реализована с помощью API вызовов, что ускоряет работу системы.

## Контейнеризация
Контейнерная виртуализация (OS Level virtualization) - использует ядро хост-системы для изоляции пользовательских процессов и ограничения ресурсов.
Рассмотрим ключевые различия.

##1. Время запуска
Для того чтобы запустить полезную нагрузку на VM нужно с помощью гипервизора выполнить инициализацию BIOS, с последующим запуском пользовательских сервисов.Это может занять некоторое количество минут.
С другой стороны, Docker путем последовательного вызова программ( docker d -> containerd -> containerd-shim -> runc -> процесс контейнера), которые под капотом совершают системные вызовы, порождает изолированный процесс, работающий в ядре хоста. Что занимает кратно меньше времени, в сравнении с VM, так как не использует лишних тяжеловесных абстракций.

##2. Размер на диске
Образ VM (ОС + базовое ПО) обычно имеет размер от нескольких ГБ.
Образ контейнера - десятки или сотни МегаБайт. Это объясняется тем, что  образ, на основе которого строится окружение контейнера, это набор последовательных директорий, где каждая последующая хранит в себе разницу с предыдущей, и хранятся там системные директории, библиотеки, исходники приложений.
Также из-за особенности структуры файловой системы контейнера UnionFS, переиспользование образа становится дешевой задачей.У нас есть Readable слои(директории, которые dockerd скачивает, либо локально собирает), Writable слой, в который записывается результат работы полезной нагрузки, Work Dir и MountDir, которая будет точкой монтирования всей этой каскадно объединенной файловой системы, и станет корневой директорией root процесса контейнера. И согласно алгоритму Copy-on-Write, в случае, если в контейнер будет вызван системный вызов на открытие и модификацию файла из Readable слоя, то эти данные будут скопированы сначала в Work Dir, а из него в Writable Layer. И если в этой фс, будет существовать несколько файлов с одинаковым именем, то использоваться будут файлы из высоких уровней, где наивысший уровень у Writable Layer. Поэтому можно поднять несколько контейнеров с одинаковым образом, и процесс поднятия будет очень быстрым, так как для создания файловой системы для этого процесса будет переиспользования кеш Readable слоев и лишь добавится собственный Writable Layer.

##3. Потребление RAM
При создании VM обычно по-умолчанию резервируется 4 ГБ, в независимости от нагрузки.
С другой стороны, контейнер это просто на уровне ядра изолированный процесс, который потербяет ровно столько ресурсов сколько нужно для работы.

##4. Уровень изоляции
VM  реализует полную изоляцию, у каждой машины есть свое ядро, свои драйверы и набор системных вызовов. И даже если удастся скомпрометировать VM, то не получится получить доступ до хост машины.
А вот если использовать некорректно настроенный образ контейнер(и его образ), можно получить доступ к root данным, будучи non-root пользователем. Также уязвимости самого ядра, могут скомпрометировать контейнер, так как он реализуется на системных вызовах хост системы.

## Почему контейнер легковесный
Нет лишнего overhead при поднятии контейнера, вся изоляция логических и физических ресурсов реализуется с помощью ядра хост ОС;
Из-за архитектуры контенейризации, контейнер не потребялет лишние физические ресурсы, а используется их как обычный процесс;
Архитектура файловой системы контейнера UnionFS с использованием драйвера overlay2 (который стоит в docker по-умолчанию) позволяет переиспользовать кеш слоев образа для поднятия новых контейнеров, тем самым ускоряя процесс создания контейнера и уменьшая объем потребляемого дискового пространства.
